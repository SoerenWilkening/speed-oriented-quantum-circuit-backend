## Phase Details

### Phase 90: Quantum Counting
**Goal**: Users can estimate the number of solutions to a search problem via quantum counting
**Depends on**: Nothing (first phase of v5.0; independent of all other v5.0 features)
**Requirements**: CNT-01, CNT-02, CNT-03
**Success Criteria** (what must be TRUE):
  1. User can call `ql.count_solutions(oracle, width=n)` and receive an integer solution count M
  2. The returned `CountResult` object exposes `.count`, `.estimate`, `.count_interval`, `.search_space`, and `.num_oracle_calls` attributes
  3. Quantum counting produces correct M for known-M oracles with M=1, M=2, and M=3 verified by Qiskit simulation
  4. `scipy>=1.10` is declared in `pyproject.toml` (previously undeclared dependency used by IQAE)
**Plans**: 2 plans

Plans:
- [ ] 90-01-PLAN.md — CountResult class, count_solutions() function, scipy dependency
- [ ] 90-02-PLAN.md — Known-M oracle verification tests (M=1, M=2, M=3)

### Phase 91: Arithmetic Bug Fixes
**Goal**: Division and modular reduction produce correct results without orphan qubits or circuit corruption
**Depends on**: Phase 90
**Requirements**: FIX-01, FIX-02, FIX-03
**Success Criteria** (what must be TRUE):
  1. Division correctly uncomputes MSB comparison temporaries -- `circuit_stats()['current_in_use']` remains stable across repeated division operations (BUG-DIV-02 fixed)
  2. QFT division and modulo produce correct results for widths 2-4 verified by Qiskit simulation (BUG-QFT-DIV fixed)
  3. Modular reduction `(a+b) mod N` produces correct results without orphan qubits -- implemented at C level or via correct Beauregard sequence replacing the broken `_reduce_mod` (BUG-MOD-REDUCE fixed)
  4. All previously-passing tests continue to pass with zero regressions
**Plans**: 3 plans

Plans:
- [x] 91-01-PLAN.md -- C-level restoring divmod + Python wiring (FIX-01, FIX-02) -- completed 2026-02-24
- [x] 91-02-PLAN.md -- C-level modular reduction + qint_mod wiring (FIX-03) -- completed 2026-02-24
- [x] 91-03-PLAN.md -- Verification, benchmarking, regression testing (FIX-01, FIX-02, FIX-03) -- completed 2026-02-24

### Phase 92: Modular Toffoli Arithmetic
**Goal**: Users can perform fault-tolerant modular arithmetic (add, sub, multiply mod N) for Shor's algorithm building blocks
**Depends on**: Phase 91 (bug fixes resolve the orphan-qubit pattern that would reproduce in Beauregard modular adder)
**Requirements**: MOD-01, MOD-02, MOD-03, MOD-04, MOD-05
**Success Criteria** (what must be TRUE):
  1. User can compute `(a + b) mod N` and `(a - b) mod N` using Toffoli gates via `qint_mod` addition and subtraction operators
  2. User can compute controlled `(a + b) mod N` inside a `with` block (controlled modular addition)
  3. User can compute `(a * c) mod N` where c is a classical integer using Toffoli gates via `qint_mod` multiplication
  4. All modular operations are verified exhaustively for widths 2-4 (statevector) and widths 5-8 (MPS simulator)
  5. Modular operations force RCA internally regardless of tradeoff policy (CLA subtraction limitation)
**Plans**: 3 plans

Plans:
- [ ] 92-01-PLAN.md -- Beauregard modular add/sub at C level (CQ + QQ + controlled variants)
- [ ] 92-02-PLAN.md -- Modular multiplication at C level + Python qint_mod rewiring
- [ ] 92-03-PLAN.md -- Exhaustive verification tests for widths 2-8

### Phase 93: Depth/Ancilla Tradeoff
**Goal**: Users can control whether the framework optimizes for circuit depth or qubit count when selecting adder implementations
**Depends on**: Phase 92 (modular arithmetic must be stable; tradeoff enhances all Toffoli operations including modular)
**Requirements**: TRD-01, TRD-02, TRD-03, TRD-04
**Success Criteria** (what must be TRUE):
  1. User can set `ql.option('tradeoff', 'auto')`, `ql.option('tradeoff', 'min_depth')`, or `ql.option('tradeoff', 'min_qubits')` to control adder selection
  2. In auto mode, CLA is selected for width >= threshold and CDKM otherwise, verified by inspecting circuit depth for small vs large widths
  3. Modular arithmetic primitives always use RCA regardless of tradeoff policy setting
  4. BK CLA subtraction limitation is documented clearly in both code docstrings and user-facing documentation
**Plans**: 2 plans

Plans:
- [x] 93-01-PLAN.md — Tradeoff option API + C-level dispatch changes + auto threshold + modular RCA verification (TRD-01, TRD-02, TRD-03)
- [x] 93-02-PLAN.md — CLA subtraction via two's complement + documentation + comprehensive verification (TRD-04, TRD-01, TRD-02)

### Phase 94: Parametric Compilation
**Goal**: Users can compile functions once and replay them with different classical values without re-capturing the gate sequence
**Depends on**: Phase 93 (tradeoff adds `tradeoff_policy` mode flag that must be in the compile cache key; all mode flags absorbed in one pass)
**Requirements**: PAR-01, PAR-02, PAR-03, PAR-04, FIX-04
**Success Criteria** (what must be TRUE):
  1. User can decorate with `@ql.compile(parametric=True)` and the function replays with different classical values without re-capture
  2. Compile cache key includes `arithmetic_mode`, `cla_override`, and `tradeoff_policy` -- switching modes invalidates cache correctly (FIX-04 fixed)
  3. Toffoli CQ operations fall back to per-value caching with clear documentation explaining value-dependent topology
  4. Oracle decorator (`@ql.grover_oracle`) forces per-value caching for structural parameters regardless of parametric setting
  5. Parametric replay produces correct circuits verified by Qiskit simulation for at least two distinct classical values
**Plans**: 3

Plans:
- [x] 94-01-PLAN.md — Mode-aware cache keys (FIX-04) + parametric API surface (PAR-01)
- [x] 94-02-PLAN.md — Parametric probe/detect/replay lifecycle (PAR-02, PAR-03)
- [x] 94-03-PLAN.md — Verification tests with Qiskit simulation (PAR-01-04, FIX-04)
