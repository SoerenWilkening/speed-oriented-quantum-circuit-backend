  <files>Backend/src/IntegerMultiplication.c</files>
  <action>
    Debug the QQ_mul function to identify and fix the segmentation fault. Suspected issues based on code analysis:

    1. **Layer count calculation**: QQ_mul uses `bits * (2 * bits + 6) - 1` for num_layer, but the actual layer usage may exceed this. Compare with cQQ_mul which uses MAXLAYERINSEQUENCE (10000).

    2. **Layer index underflow**: The code uses `layer -= bits - rounds` in all_rot() which can decrement layer. If layer goes negative (num_t is unsigned), this causes massive overflow leading to out-of-bounds access.

    3. **Qubit index calculation**: Helper functions use expressions like `3 * bits + bit_int2 - 1` where bit_int2 can be negative. Verify this doesn't produce invalid qubit indices.

    4. **Gate array allocation**: Each layer is allocated with `calloc(2 * bits, sizeof(gate_t))` but the actual gate count per layer may exceed 2*bits for certain operations.

    Debugging steps:
    - Add bounds checking to QQ_mul to detect where the crash occurs
    - Compare layer usage with num_layer allocation
    - Verify qubit indices stay within valid range (0 to 3*bits-1 for QQ_mul)
    - Check if gates_per_layer increments exceed array size

    Fix approach: Once root cause identified, either:
    - Increase num_layer allocation (like cQQ_mul does with MAXLAYERINSEQUENCE)
    - Fix layer arithmetic to prevent underflow
    - Increase gate array sizes per layer
    - Add bounds checks with meaningful error messages

    Compare with working CQ_mul to understand the expected behavior.
