  <name>Task 1: Create verification script with framework, CLI, and all test cases</name>
  <files>scripts/verify_circuit.py</files>
  <action>
Create `scripts/verify_circuit.py` as a standalone self-contained script. Structure:

**1. Imports and constants:**
- `import quantum_language as ql` for circuit building and QASM export
- `import qiskit.qasm3` for QASM 3.0 loading
- `from qiskit_aer import AerSimulator` for deterministic simulation
- `import argparse, sys, os` for CLI and output
- ANSI color constants gated on `should_use_colors()` (check `sys.stdout.isatty()`, `NO_COLOR` env var, `CLICOLOR_FORCE` env var)

**2. TestCase dataclass:**
```python
@dataclass
class TestCase:
    name: str
    category: str
    description: str
    expected: int          # Expected integer result
    width: int             # Bit width of result register
    qasm_generator: Callable[[], str]  # Builds circuit, returns QASM string
```

**3. Test case factory functions:**
Each factory function calls `ql.circuit()` to reset, creates qint operands with explicit width, performs the operation, and returns `ql.to_openqasm()`.

**4. ArithmeticTests class** (static methods returning list of TestCase):
- addition_basic: 3 + 5 = 8 (4-bit)
- addition_overflow: 14 + 5 = 3 (4-bit, wraps mod 16). Note: addition result width is max(a.width, b.width) + 1, so use appropriate widths. If the ql library produces a 5-bit result for 4-bit inputs, then 14+5=19 fits in 5 bits with no overflow. Adjust test values and expected results based on actual ql behavior: test with values that truly overflow at the result width. IMPORTANT: Check how `qint.__add__` determines result width. If result is always wider, pick values that overflow at THAT width.
- subtraction_basic: 7 - 3 = 4 (4-bit)
- subtraction_underflow: 3 - 7 = 12 (4-bit, wraps mod 16). Same width consideration as addition — check actual result width from ql.
- multiplication_basic: 3 * 4 = 12 (4-bit operands). Result width for multiplication is typically a.width + b.width, so 8-bit result. Expected = 12.
- multiplication_overflow: Pick values where product overflows result width. If result is a.width + b.width = 8 bits, use e.g. 15 * 20 with appropriate widths.

**5. ComparisonTests class** (static methods returning list of TestCase):
Comparisons return qbool (1-bit): 1 for true, 0 for false.
- less_than_true: 3 < 7 → 1 (width=1)
- less_than_false: 7 < 3 → 0 (width=1)
- less_equal_true: 5 <= 5 → 1 (width=1)
- less_equal_false: 7 <= 3 → 0 (width=1)
- equal_true: 5 == 5 → 1 (width=1)
- equal_false: 3 == 7 → 0 (width=1)
- greater_equal_true: 7 >= 5 → 1 (width=1)
- greater_equal_false: 3 >= 7 → 0 (width=1)
- greater_than_true: 7 > 3 → 1 (width=1)
- greater_than_false: 3 > 7 → 0 (width=1)
- not_equal_true: 3 != 7 → 1 (width=1)
- not_equal_false: 5 != 5 → 0 (width=1)

**6. BitwiseTests class** (static methods returning list of TestCase):
Use 4-bit qints.
- and_basic: 0b1100 & 0b1010 = 0b1000 = 8 (width=4)
- or_basic: 0b1100 | 0b1010 = 0b1110 = 14 (width=4)
- xor_basic: 0b1100 ^ 0b1010 = 0b0110 = 6 (width=4)
- not_basic: ~0b1010 = 0b0101 = 5 (width=4)

**7. Verification engine function `run_verification()`:**
- Takes list of TestCase, fail_fast, show_qasm, log_file, category filter
- Creates AerSimulator(method='statevector') once
- For each test:
  - Call test.qasm_generator() to get QASM string (wrapped in try/except)
  - Load via qiskit.qasm3.loads(qasm_str)
  - Check circuit.cregs; if empty, call circuit.measure_all()
  - Run simulator with shots=1
  - Extract bitstring from counts, convert via int(bitstring, 2)
  - **Bit extraction:** The result register may not be the only register. The verification needs to extract the correct bits for the result. Document the approach: if the QASM has named registers, extract by register name; otherwise use full bitstring with appropriate bit slicing based on test.width.
  - Compare actual vs expected
  - Print dot (pass) or F (fail) or E (error) with ANSI colors
  - Collect failures with details
- After all tests: print summary line
- Print detailed failure report (expected vs actual, and QASM snippet if show_qasm)
- If log_file specified, write detailed log
- Return 0 if all pass, 1 if any fail

**8. CLI with argparse:**
- `--fail-fast`: stop on first failure
- `--category {arithmetic,comparison,bitwise}`: filter tests
- `--show-qasm` / `-v`: show QASM in failure output
- `--log PATH`: write log file

**9. `collect_all_tests()` function:**
Gathers tests from all three category classes into a single list.

**10. `if __name__ == '__main__':` entry point calling main().**

**Critical implementation notes:**
- Each test's qasm_generator MUST call `ql.circuit()` first to reset the circuit state before building
- Use `_ =` prefix for intermediate quantum variables that are used for side effects only
- Result extraction from bitstring is the trickiest part. Qiskit returns measurement results keyed by classical register. When measure_all() is used, all qubits are measured. The result qubit(s) of interest are at specific positions. The script must extract the right bits. Strategy: use the full bitstring and extract the last `width` bits (LSB) if the result variable is the most recently allocated. HOWEVER, this depends on qubit allocation order in the ql library. A safer approach: after getting the bitstring, try to match by register if named registers exist, or document the limitation.
- The ql library allocates qubits sequentially. For `a + b`, qubit allocation is: a's qubits, b's qubits, result's qubits (and possibly ancilla). The measurement bitstring from measure_all() will have ALL qubits. The result bits are at a specific offset. This is complex to extract generically. **Recommended approach:** Since we know the circuit structure (we built it), we know the expected qubit positions. But a simpler approach: measure ONLY the result qubits by examining the circuit. Or: compare the full measurement result by computing what the full bitstring should be (including input values in their registers). This is deterministic so we can predict the full state.
- **Simplest correct approach for verification:** Instead of extracting partial results, predict the FULL measurement outcome (all qubits) and compare. For `a=3 (4-bit) + b=5 (4-bit) = result (5-bit)`: full state is a=3, b=5, result=8, plus any ancilla in known state. But ancilla state depends on uncomputation behavior.
- **Alternative simplest approach:** Use the register-based extraction from Qiskit counts. When QASM declares named registers (`qubit[n] q;`), measure_all creates classical registers. Qiskit counts keys use space-separated register values. Check the Qiskit counts format and extract accordingly.
- Given complexity of bit extraction, add a helper function `extract_result(counts, test)` that handles the extraction logic. If extraction proves unreliable, fall back to comparing the full integer value of the bitstring against a known-good expected full-state value. Document the approach chosen.
