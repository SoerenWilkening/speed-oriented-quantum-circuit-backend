Create `tests/test_array_verify.py` that verifies ql.array operations through the full pipeline.

**CRITICAL: BUG-ARRAY-INIT discovered in research.**
The `ql.array([1, 2], width=3)` constructor has a bug: it passes `self._width` as the VALUE to `qint()` instead of as the width parameter. This means array elements get initialized to wrong quantum values. See RESEARCH.md "Array Constructor Bug" section for details.

**Workaround strategy:** Construct arrays by creating individual `qint(value, width=w)` objects manually and then using `ql.array()` from those objects. If `ql.array` can accept a list of qint objects, use that. If not, test what `ql.array` does produce and xfail tests that require correct value initialization.

**Step 1: Calibration tests (run first to understand actual behavior).**

Write a calibration function that:
1. Creates an array using `ql.array([values], width=w)`
2. Also creates the same array manually: individual `qint(v, width=w)` objects
3. Runs both through the pipeline, prints full bitstrings
4. Determines extraction positions for array results

`test_array_calibration_constructor`: Create `ql.array([1, 2], width=3)`, export QASM, print it. Then create `a=ql.qint(1,width=3); b=ql.qint(2,width=3)`, export QASM, compare. This reveals BUG-ARRAY-INIT empirically.

`test_array_calibration_sum`: Create two qints manually, sum them via the array API if possible, or via direct addition. Print bitstring to determine result position.

**Step 2: Array reduction tests.**

Given BUG-ARRAY-INIT, there are two approaches:
- **Approach A (preferred):** If `ql.array` can wrap existing qint objects, create qints manually then wrap. Test reductions on the wrapper.
- **Approach B (fallback):** If wrapping doesn't work, xfail all array tests with reason="BUG-ARRAY-INIT" and document what happens.

For each reduction, test with 2-element and single-element arrays:

**Sum reduction:**
- `test_array_sum_2elem`: Array of [1, 2] (width=3). Expected sum = 3.
- `test_array_sum_1elem`: Array of [3] (width=3). Expected sum = 3 (identity).
- `test_array_sum_overflow`: Array of [3, 3] (width=2). Expected sum = (3+3)%4 = 2 (overflow wrapping).

**AND reduction (.all()):**
- `test_array_and_2elem`: Array of [3, 1] (width=2). Expected AND = 3 & 1 = 1.
- `test_array_and_1elem`: Array of [3] (width=2). Expected AND = 3 (identity).

**OR reduction (.any()):**
- `test_array_or_2elem`: Array of [1, 2] (width=2). Expected OR = 1 | 2 = 3.
- `test_array_or_1elem`: Array of [2] (width=2). Expected OR = 2 (identity).

**Step 3: Element-wise operation tests.**

- `test_array_add_scalar`: Array [1, 2] + scalar 1. Expected: [2, 3]. Extract last element = 3.
- `test_array_sub_scalar`: Array [3, 2] - scalar 1. Expected: [2, 1]. Extract last element = 1.

**Pipeline approach:**
- Use the `verify_circuit` fixture where possible (result = last allocated register)
- For reductions, the result is a new qint from the reduction operation — it should be the last allocated and thus at `bitstring[:width]`
- For element-wise ops, the result array elements are last allocated
- If extraction positions are unclear, use calibration (print bitstring, find the known value)

**xfail strategy:**
- If `ql.array` constructor bug prevents correct initialization: `@pytest.mark.xfail(reason="BUG-ARRAY-INIT", strict=False)`
- If array operations themselves have issues beyond initialization: document as new bug
- All xfail markers: non-strict with descriptive reason

**Implementation details:**
- `import gc, warnings, pytest`
- `warnings.filterwarnings("ignore", message="Value .* exceeds")`
- Width 2-3 bits as specified in CONTEXT.md
- Array sizes 1-3 elements as specified in CONTEXT.md
- Return keepalive refs from all circuit builders
- If ql.array accepts qint list: `arr = ql.array([ql.qint(1, width=3), ql.qint(2, width=3)])` — try this first
- Module docstring explaining VADV-03 and BUG-ARRAY-INIT workaround
