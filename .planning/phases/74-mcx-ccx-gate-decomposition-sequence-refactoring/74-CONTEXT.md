# Phase 74: MCX/CCX Gate Decomposition & Sequence Refactoring - Context

**Gathered:** 2026-02-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Decompose all MCX gates (3+ controls) into CCX/CX/X gates automatically, add opt-in CCX->Clifford+T decomposition via `toffoli_decompose` option, generate dedicated decomposed hardcoded sequences, and refactor overly large C files for maintainability. No new arithmetic operations or algorithms.

</domain>

<decisions>
## Implementation Decisions

### MCX auto-decomposition scope
- All MCX gates with 3+ controls are decomposed everywhere, not just specific contexts
- Always-on when `fault_tolerant=True` -- no separate option needed for MCX decomposition
- Support arbitrary control counts via recursive AND-ancilla decomposition (no cap)
- Use the same AND-ancilla pattern from Phase 72-03 (CCX compute + CCX uncompute) for all MCX
- Extend the inline decomposition pattern from multiplication to all MCX-producing codepaths (controlled add, CLA, etc.) so MCX gates never appear in output
- Claude audits codebase to find every MCX emission point and ensures coverage
- Remove MCX field from gate_counts/ql.stats() since MCX will never appear in Toffoli mode output

### CCX->Clifford+T API design
- Separate option: `ql.option('toffoli_decompose', True)` independent from `fault_tolerant`
- Works independently of `fault_tolerant` -- decomposes any CCX in the circuit regardless of arithmetic backend
- Standard decomposition: 7 T-gates + 6 CNOTs (exact, not relative-phase)
- Target basis: {H, T, Tdg, CX} -- CX (CNOT) is the native 2-qubit gate, no further decomposition
- Add H, T, Tdg as new gate enum values, reusing existing gate infrastructure for counting/visualization/QASM export
- When `toffoli_decompose` is on, count actual T/Tdg gates in the circuit (not estimation formula)

### Decomposition timing
- MCX decomposition: inside sequence-generating C functions (each function emits CCX directly instead of MCX)
- CCX->Clifford+T decomposition: inside sequence generators (check `toffoli_decompose` flag, emit H/T/Tdg/CX instead of CCX)
- Hardcoded sequences: separate decomposed files generated by script (not load-time decomposition)
- Ancilla allocation for AND-ancilla: static indices in generation script (dispatch knows exact qubit count upfront)

### File refactoring
- Dedicated refactoring plan FIRST (pure refactor, no behavior change) before decomposition work
- Refactor everything: both algorithm implementation files AND dispatch routers
- Keep flat structure in sequences/ (no subdirectories) -- prefix naming handles disambiguation
- Keep generation scripts separate per backend (QFT, Toffoli, decomposed)
- Pain points to address: ToffoliAddition.c and hot_path_add.c / hot_path_mul.c specifically
- Claude's discretion on organizing principle for splitting C files (by algorithm, variant, or operation)

### Claude's Discretion
- Organizing principle for C file splits (by algorithm vs variant vs operation)
- Whether headers need splitting (based on size and dependency graph)
- Internal function naming conventions for decomposed variants
- Test structure and organization

</decisions>
