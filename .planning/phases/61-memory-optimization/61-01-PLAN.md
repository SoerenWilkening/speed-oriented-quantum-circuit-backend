<task type="auto">
  <name>Task 1: Create memory profiling script and run memray at three widths</name>
  <files>scripts/profile_memory_61.py</files>
  <action>
Create `scripts/profile_memory_61.py` -- a memory profiling script that exercises the three hot path operations (add, mul, xor) at configurable bit widths. The script should:

1. Accept a command-line argument for bit width (default: 8)
2. Accept an optional `--iterations` argument (default: 100)
3. Run three phases:
   - Addition hot path: create circuit, allocate two qints, run `a += b` for N iterations
   - Multiplication hot path: create circuit, allocate two qints, run `a * b` for N iterations (use pedantic setup to create fresh circuit each time since mul allocates qubits)
   - XOR hot path: create circuit, allocate two qints, run `a ^= b` for N iterations
4. Print sizeof(gate_t) by importing from the C extension if possible, or note it for manual verification

Run memray profiling at all three widths. Execute from the project root with the virtual environment activated:

```bash
# Build the package first (ensure latest C code is compiled)
. .venv/bin/activate && pip install -e . -q

# Run memray at three widths
PYTHONPATH=src memray run --native -o /tmp/memray_61_8bit.bin -- python scripts/profile_memory_61.py 8
PYTHONPATH=src memray run --native -o /tmp/memray_61_16bit.bin -- python scripts/profile_memory_61.py 16
PYTHONPATH=src memray run --native -o /tmp/memray_61_32bit.bin -- python scripts/profile_memory_61.py 32

# Generate flame graphs
memray flamegraph /tmp/memray_61_8bit.bin -o /tmp/memray_61_8bit.html
memray flamegraph /tmp/memray_61_16bit.bin -o /tmp/memray_61_16bit.html
memray flamegraph /tmp/memray_61_32bit.bin -o /tmp/memray_61_32bit.html

# Generate stats output (allocation counts and sizes)
memray stats /tmp/memray_61_8bit.bin > /tmp/memray_61_8bit_stats.txt
memray stats /tmp/memray_61_16bit.bin > /tmp/memray_61_16bit_stats.txt
memray stats /tmp/memray_61_32bit.bin > /tmp/memray_61_32bit_stats.txt
```

If `memray --native` fails on the platform, retry without `--native` (still captures Python-level malloc through C extensions).

Note: Per research, expect to see:
- run_instruction() malloc (leaked, ~40 bytes per gate)
- reverse_circuit_range() malloc (leaked, same pattern)
- colliding_gates() malloc (24 bytes per add_gate call, properly freed)
- Sequence creation mallocs (one-time, cached)
