    This task fixes BUG-BIT-01 for CQ (classical-quantum) bitwise operations. There are two sub-bugs to fix:

    **Sub-bug A: CQ AND and CQ OR produce wrong results**

    First, diagnose by running a single failing CQ test manually to understand the exact failure mode. Example: `qint(3, width=2) & 1` should produce 1 (binary 01), but may produce wrong value.

    The C backend functions `CQ_and()` and `CQ_or()` in LogicOperations.c (lines 1070-1309) use `two_complement(value, bits)` which returns bits in MSB-first order (bin[0]=MSB, bin[bits-1]=LSB). The gate indices then use `i` as both the bit position and the qubit_array index.

    The Python layer in qint_bitwise.pxi populates the qubit_array as:
    - qubit_array[0..result_bits-1] = result qubits (MSB-first from result.qubits[64-result_bits:64])
    - qubit_array[result_bits..result_bits+self.bits-1] = self qubits (MSB-first)

    Investigate whether the bit ordering between two_complement and the Python qubit_array is consistent. If two_complement bin[0] is MSB and qubit_array[0] is MSB, the CNOT at target=i, control=bits+i should be correct.

    If the C-level logic is actually correct, the bug may be in the Python layer -- for example, when `result_bits > self.bits` in the CQ case (classical value has more bits than quantum operand), the qubit_array has uninitialized entries between self's qubits and the end.

    Check if `result_bits` can exceed `self.bits` in the CQ path. When `other` is an int:
    ```python
    classical_width = other.bit_length() if other > 0 else 1
    result_bits = max(self.bits, classical_width)
    ```
    If `classical_width > self.bits`, then `result_bits > self.bits` and the qubit_array is not fully populated for the C function. Fix this by zero-extending self's qubits.

    Also verify: when `self.bits == result_bits` (the common same-width case), are there still failures? If yes, the C function itself has a bug. Run specific test cases to isolate.

    Fix approach (adapt based on diagnosis):
    1. If C functions are buggy: fix the gate index calculation in CQ_and/CQ_or
    2. If Python qubit_array population is wrong: fix the zero-extension of self when result_bits > self.bits
    3. Both fixes may be needed

    **Sub-bug B: CQ XOR uses per-bit Q_not(1) loop**

    The __xor__ method in qint_bitwise.pxi (lines 342-363) does NOT call a CQ_xor C function. Instead it loops through bits applying Q_not(1) for each set bit of the classical value. This approach may have qubit addressing issues.

    Check if the per-bit X gate application correctly addresses `result.qubits[64 - result_bits + (result_bits - 1 - i)]`. The index calculation maps bit `i` (from LSB) to the correct physical qubit. Verify this is correct and fix if not.

    Optionally: create a `CQ_xor` C function (similar to CQ_and/CQ_or) and expose it via _core.pxd and use it in qint_bitwise.pxi for consistency and correctness. This is simpler than the per-bit loop and more likely to be correct.

    After fixes, rebuild the package: `pip install -e .` or `python3 setup.py build_ext --inplace`

  <action>
    Fix mixed-width bitwise operations in qint_bitwise.pxi. The issue: when operand widths differ, the qubit_array is not populated correctly for the C function.

    **Root cause in __and__ (line 72-100):**
    ```python
    result_bits = max(self.bits, other.bits)
    # Output qubits
    qubit_array[:result_bits] = result.qubits[result_offset:64]
    # Self qubits -- ONLY self.bits entries, but C expects result_bits entries
    qubit_array[result_bits:result_bits + self.bits] = self.qubits[self_offset:64]
    # Other qubits -- ONLY other.bits entries, but C expects result_bits entries
    qubit_array[2*result_bits:2*result_bits + other.bits] = other.qubits[other_offset:64]
    ```

    When `self.bits < result_bits`, `Q_and(result_bits)` expects `result_bits` qubits for operand A at positions [result_bits, 2*result_bits-1], but only `self.bits` qubits are placed. Positions [result_bits+self.bits, 2*result_bits-1] contain stale/uninitialized values from a previous qubit_array use, causing wrong qubit mappings and qubit allocation overflow.

    **Fix for __and__:**
    When self.bits < result_bits (or other.bits < result_bits), the narrower operand's qubits must be zero-extended. The "zero" qubits are ancilla qubits in |0> state. Two approaches:

    Approach A (preferred): Allocate a temporary zero-extended qint for the narrower operand:
    ```python
    if self.bits < result_bits:
        # Create zero-extended copy: copy self qubits into a wider qint
        # The extra high bits are already |0> (ancilla)
        extended_self = qint(width=result_bits)
        # Copy self bits into extended_self using XOR (extended_self ^= self)
        # This requires running Q_xor with self.bits width
        ...
    ```

    Approach B (simpler): For QQ, instead of calling Q_and/Q_or with result_bits, call with min(self.bits, other.bits) and handle the extra bits separately. For AND, any bit where either operand is 0 produces 0 (already in |0> state). So only the overlapping bits need Toffoli gates.

    Approach C (most pragmatic): Allocate ancilla qubits for the missing positions and place them in the qubit_array. Since ancilla starts at |0>, AND with |0> = |0> (correct), OR with |0> = other bit (needs CNOT copy), XOR with |0> = other bit (needs CNOT copy).

    **Recommended approach for each operation:**

    For AND: The narrower operand's higher bits are logically 0 (zero-extension). 0 AND x = 0. So the result's higher bits should remain |0>. Solution: only apply Q_and for the overlapping `min_bits = min(self.bits, other.bits)` range, then copy the remaining bits from the wider operand for OR/XOR (not needed for AND since 0 AND x = 0).

    Actually, the simplest correct fix: ensure the qubit_array has valid (|0> state) qubits for all positions the C function will access. Allocate ancilla qubits for the gap:
    ```python
    if self.bits < result_bits:
        padding = result_bits - self.bits
        pad_start = allocator_alloc(allocator, padding, True)  # ancilla = |0>
        # Place self qubits at MSB positions, padding at LSB
        # Or: place padding at the high-bit positions (depends on bit ordering)
    ```

    Wait -- since qint stores bits MSB-first and the narrower operand is zero-extended at the MSB (high bits are 0), the padding should go at the START of the operand's section in qubit_array:

    ```python
    # For self with fewer bits: zero-extend at high positions
    gap = result_bits - self.bits
    # Positions [result_bits, result_bits+gap-1]: ancilla |0> qubits (high bits)
    # Positions [result_bits+gap, 2*result_bits-1]: self's actual qubits (low bits)
    ```

    Actually, re-examine: the qubit_array maps abstract index to physical qubit. Index 0 is MSB for the C function. If self has 2 bits and result needs 3 bits, the C function expects operand A at indices [3,4,5] (for result_bits=3). Index 3 is MSB of operand A. Self's MSB should map to index 3 only if self is 3-bit. If self is 2-bit, self's MSB maps to index 4 and index 3 needs a |0> ancilla.

    **Precise fix for __and__ (QQ case):**
    ```python
    # Operand A (self) -- zero-extend at MSB positions
    if self.bits < result_bits:
        gap = result_bits - self.bits
        # Allocate gap ancilla qubits (they start in |0>)
        pad_start = allocator_alloc(circuit_get_allocator(<circuit_s*>_circuit), gap, True)
        for j in range(gap):
            qubit_array[result_bits + j] = pad_start + j
        qubit_array[result_bits + gap:result_bits + gap + self.bits] = self.qubits[self_offset:64]
    else:
        qubit_array[result_bits:result_bits + self.bits] = self.qubits[self_offset:64]
    ```

    Apply the same pattern to operand B (other) and to __or__ and __xor__ methods.

    For CQ mixed-width: similar issue -- when result_bits > self.bits, self needs zero-extension in the qubit_array. Apply the same padding logic.

    IMPORTANT: Check how the qubit allocator is accessed from the .pxi file. It may need to go through `circuit_get_allocator`. Look at how other parts of the codebase allocate ancilla.

    After making all fixes, rebuild: `pip install -e .` or equivalent.
