**Step 2: If tests still fail, investigate Python __sub__ implementation.**

The `__sub__` method (qint.pyx line 824-854) does out-of-place subtraction:
```python
a = qint(value=self.value, width=result_width)
a -= other
return a
```

This creates a NEW qint initialized with `self.value` (the classical init value), then subtracts `other` in-place. The issue is that `self.value` may not match the current quantum state if `self` has been modified by prior operations. However, for fresh qints (like in the bug tests), `self.value` should be correct.

**If subtraction still fails after BUG-04 fix:**
The problem is likely in the `addition_inplace` method with `invert=True`. Check:
1. Does `invert=True` properly negate the operation? (line 711: `run_instruction(seq, &arr[0], invert, _circuit)`)
2. Is the inversion applied at the C level correctly?
3. For out-of-place: does `qint(value=self.value)` correctly initialize the quantum register?

**If the issue is that out-of-place creates wrong initial state:**
The `__sub__` creates `a = qint(value=self.value, width=result_width)`. This initializes a brand new quantum register with the classical value. Then `a -= other` subtracts. For `3 - 7`: creates qint(3), then subtracts 7 in-place, should get 3-7 = -4 mod 16 = 12.

If instead the result is 7, that would mean the subtraction produced `self.value - other = 7` which is `|7 - 7| = 0` or some misinterpretation. This points to `self.value` being wrong or the subtraction direction being reversed.

**Potential fix if needed:** If `__sub__` computes `other - self` instead of `self - other`, the inversion logic is reversed. Check whether `a -= other` with `a` initialized to `self.value` does `a = a - other` or `a = other - a`.

In `__isub__` (line 856-876): `self.addition_inplace(other, invert=True)`. This calls `CQ_add(self.bits, other_value)` with `invert=True`. The `invert` flag in `run_instruction` inverts the entire sequence. If CQ_add computes `target += value`, then inverted it computes `target -= value`. So `a -= 7` should compute `a = a - 7 = 3 - 7 = -4 mod 16 = 12`. This should be correct IF the addition is correct.

**If all subtraction tests pass after BUG-04 fix:** No changes needed -- the subtraction was always correct, just blocked by broken QFT arithmetic.

**Step 2: If comparison tests still fail, investigate `__le__` implementation.**

The `__le__` method (qint.pyx line 1901-1967) for qint operands:
1. `self -= other` (in-place subtraction)
2. `is_negative = self[64 - self.bits]` (check MSB)
3. `is_zero = (self == 0)` (check if zero)
4. `result = qbool(); result ^= is_negative` (init result to is_negative)
5. `temp_zero = qbool(); temp_zero ^= is_zero` (copy is_zero)
6. `result |= temp_zero` (OR: result = is_negative | is_zero)
7. `self += other` (restore)

For `5 <= 5`:
- `self -= other` -> 5 - 5 = 0
- MSB check: 0 is not negative -> is_negative = 0
- Zero check: 0 == 0 -> is_zero = 1
- Result = 0 | 1 = 1 (True)

This logic is correct. If it still fails, the issue is in:
- The subtraction step (BUG-01 dependency)
- The MSB extraction `self[64 - self.bits]`
- The equality check `self == 0`
- The qbool XOR/OR operations
