**Background:** CQ_mul multiplies a quantum register by a classical value using QFT-based arithmetic. The current implementation (lines 72-150) applies QFT, then controlled-phase rotations, then inverse QFT. Plan 29-05 applied a bit-ordering fix (`bin[bits-1-bit_int2]`) but results still return 0 for all inputs.

**Step 1: Understand QFT multiplication algorithm**

QFT multiplication of quantum register `|a>` by classical value `c`:
1. Apply QFT to `|a>` (transforms to Fourier basis)
2. In Fourier basis, multiplication by `c` requires phase rotations encoding `c * a`
3. Apply inverse QFT

The key insight: In the Fourier basis, the k-th qubit has phase proportional to `a * 2^k`. To multiply by `c`, each qubit's phase must be rotated by an amount proportional to `c`.

For CQ_mul, the phase rotation for target qubit `target` from control qubit `control` (which represents bit `bit` of the quantum input) should encode the product. The total phase accumulated on qubit `j` should be:

`phase_j = 2*PI * (c * a) / 2^(bits-j)`

Since `a` is in the quantum register (control qubits), and `c` is classical:

`phase_j = sum_k (a_k * c_k_contribution_to_qubit_j)`

**Step 2: Analyze current CQ_mul code (lines 120-143)**

```c
for (int bit = bits - 1; bit >= 0; --bit) {
    layer = 2 * bits + 2 * rounds - 1;
    num_t control = bits + bit;
    for (int i = 0; i < bits - rounds; ++i) {
        num_t target = bits - i - 1 - rounds;
        double value = 0;
        for (int bit_int2 = 0; bit_int2 < bits; ++bit_int2) {
            value += bin[bits - 1 - bit_int2] * 2 * M_PI / (pow(2, i + 1)) * pow(2, bit_int2);
        }
        gate_t *g = &mul->seq[layer][mul->gates_per_layer[layer]++];
        cp(g, target, control, value);
        layer++;
    }
    rounds++;
}
```

The value computation sums over all bits of the classical value:
`value = sum_{bit_int2} bin[bits-1-bit_int2] * 2*PI / 2^(i+1) * 2^(bit_int2)`

This is: `value = c_decimal * 2*PI / 2^(i+1)` where `c_decimal = sum bin[bits-1-bit_int2] * 2^bit_int2` is the classical value.

Wait -- this means the `value` is the SAME for every iteration of the inner loop (since the sum over bit_int2 gives the full classical value). The only thing varying is `i` which changes the power of 2 divisor and the target qubit.

**Step 3: Trace through 2-bit example**

For CQ_mul(2, 3) (multiply quantum register by 3):
- `two_complement(3, 2)` returns `bin = [1, 1]` (MSB-first: bit1=1, bit0=1)
- With fix: `bin[bits-1-bit_int2]` = bin[1-bit_int2]
  - bit_int2=0: bin[1] = 1 (LSB, weight 1)
  - bit_int2=1: bin[0] = 1 (MSB, weight 2)
  - c_decimal = 1*1 + 1*2 = 3 ✓

QFT on qubits 0,1 of the target register. Control qubits at indices 2,3 (second register).

Round 0 (bit=1, rounds=0):
- control = 2+1 = 3 (qubit 3 of input register, MSB)
- i=0: target = 2-0-1-0 = 1, value = 3 * 2*PI/2 = 3*PI
- i=1: target = 2-1-1-0 = 0, value = 3 * 2*PI/4 = 3*PI/2
rounds=1

Round 1 (bit=0, rounds=1):
- control = 2+0 = 2 (qubit 2 of input register, LSB)
- i=0: target = 2-0-1-1 = 0, value = 3 * 2*PI/2 = 3*PI
rounds=2

So we get:
- CP(target=1, control=3, phase=3*PI)
- CP(target=0, control=3, phase=3*PI/2)
- CP(target=0, control=2, phase=3*PI)

**Step 4: What should the correct circuit be?**

For QFT multiplication of `|a>` by classical `c=3`:
- In Fourier basis, qubit `a_j` has phase proportional to `a * 2^j / 2^bits`
- After multiplication by c, qubit `a_j` should have phase proportional to `c * a * 2^j / 2^bits`
- This means each control-phase gate from `a_k` to target `j` should have phase:
  `2*PI * c * 2^k / 2^(j-k+1+bits)` -- WAIT, this is getting confused.

**Let me reconsider.** QFT multiplication by a classical constant is equivalent to repeated QFT addition. To multiply `|a>` by `c`, you need to add `a` to a result register `c` times, or equivalently add `c*2^k` for each bit k of `a`.

Actually, looking at the code structure more carefully: CQ_mul has target qubits [0, bits-1] and control qubits [bits, 2*bits-1]. The control qubits represent the INPUT quantum value `a`, and the target qubits will hold the RESULT `c*a`.

But wait -- the target register starts at 0 and after QFT it's in Fourier basis. The result is written TO the target register. But what was in the target register initially? Looking at how qint multiplication works in qint.pyx, the target register should be initialized to 0 (an accumulator).

**The issue might be that the value computation is wrong.** Let me compare with how CQ_add works:

CQ_add encodes a classical value `c` as phase rotations on the target register (already in Fourier basis from QFT). For CQ_mul, we need to encode `c * a_bit * 2^bit_position` for each bit of the quantum input `a`.

For each control bit `a_k` (at position `bit` in the code), the controlled-phase rotation should add `c * 2^k` to the Fourier-basis target register. The phase for target qubit `j` from control `a_k` adding `c * 2^k` is:

`phase = 2*PI * c * 2^k / 2^(bits - j)`

Hmm, let me simplify. In QFT representation, the phase on qubit `j` (where j=0 is the qubit that gets smallest rotation) to add a value `V` is:

For qubit j, phase = `2*PI * V / 2^(bits-j)` ... No.

Actually, in the standard QFT addition formulation used by CQ_add:
- Target qubit `qubit` (0 to bits-1) receives from bit `bit_idx` of the classical value:
  `phase = c_bit * 2*PI / 2^(qubit - bit_idx + 1)` for qubit >= bit_idx

For multiplication, control qubit `a_k` (k-th bit of quantum input) applies a conditional addition of `c * 2^k`. So the controlled phase from `a_k` to target qubit `j` should be:

`phase = sum over bit_idx of (c_bit_idx * 2*PI / 2^(j - bit_idx - k + 1))`
where we're adding the value `c * 2^k` (classical value c shifted left by k).

Equivalently, this is the CQ_add rotation formula but with an extra `* 2^k` scaling: each rotation is `2^k` times larger because we're adding `c * 2^k` instead of just `c`.

**Now compare with the code:**
The code has: `value += bin[bits-1-bit_int2] * 2*PI / pow(2, i+1) * pow(2, bit_int2)`

Here `i` maps to target qubit position and `bit_int2` maps to bit position of the classical value. The `pow(2, bit_int2)` factor gives the weight of each classical bit. But there's NO `pow(2, k)` factor for the control bit position!

**THIS IS THE BUG.** The value should include a factor for the control bit's positional weight. Each control bit `a_k` should multiply the total phase by `2^k`:

Change:
```c
value += bin[bits - 1 - bit_int2] * 2 * M_PI / (pow(2, i + 1)) * pow(2, bit_int2);
```
To:
```c
value += bin[bits - 1 - bit_int2] * 2 * M_PI / (pow(2, i + 1)) * pow(2, bit_int2) * pow(2, bit);
```

Wait -- but `bit` goes from `bits-1` down to 0, and represents the INDEX in the second register. Does `bit` = k (the bit position value)?

If qubit `bits + bit` represents the `bit`-th bit of the quantum input (with bit=0 = LSB), then yes, the weight is `2^bit`.

BUT -- same issue as QQ_add: is `bit` correctly mapping to the bit position? The control qubit at index `bits + bit` with bit going from bits-1 to 0 -- if the quantum register is laid out LSB-first (qubit bits = LSB, qubit 2*bits-1 = MSB), then `bit` directly gives the bit position. But we saw in QQ_add that this mapping was WRONG (control bits were swapped).

**Combined fix approach:**
1. Add the missing `pow(2, bit)` factor for the control bit weight
2. Also consider if the control bit mapping needs reversal (as discovered in QQ_add investigation)

**If control mapping is correct (bit=LSB index):**
```c
value += bin[bits - 1 - bit_int2] * 2 * M_PI / (pow(2, i + 1)) * pow(2, bit_int2) * pow(2, bit);
```

**If control mapping is reversed (like QQ_add):**
```c
value += bin[bits - 1 - bit_int2] * 2 * M_PI / (pow(2, i + 1)) * pow(2, bit_int2) * pow(2, bits - 1 - bit);
```

**Testing approach:** Try option 1 first. If multiplication produces non-zero but wrong results, try option 2.

**Also fix the target qubit mapping if needed.** The target qubits use `target = bits - i - 1 - rounds` which reverses as i increases. Check if this matches the QFT phase ordering. Looking at CQ_add for reference: CQ_add applies phase to qubit `i` directly (line 115: `p(..., i, rotations[i])`). But CQ_mul uses `bits - i - 1 - rounds` which is different. This target indexing may also need correction.

**Practical approach:**
