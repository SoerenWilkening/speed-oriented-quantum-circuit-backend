**BACKGROUND:** Plan 29-09 fixes QQ_add by changing `target = bits - i - 1 - rounds` to `target = rounds + i`. The same QFT addition pattern is used inside multiplication functions via the CP_sequence helper and directly in CQ_mul block 1.

**Fix 1: CP_sequence (line 24)**

Current:
```c
num_t target = bits - i - 1 - rounds;
```

Change to:
```c
num_t target = rounds + i;
```

CP_sequence is called from QQ_mul block 1 (line 215), all_rot (line 64), and cQQ_mul. The same Draper QFT target mapping applies: for a given rounds (bit position), the target qubits should start at `rounds` and increase, not start at `bits-1-rounds` and decrease.

**IMPORTANT: Check the `inverted` path in CP_sequence.** The function has an inverted mode where it iterates `l1=bits-1, l2=0, fac=-1`. When inverted:
- The loop goes `i = bits-1 down to 0` with `fac = -1`
- Wait, this is wrong — `for (int i = l1; i < l2; i += fac)` with l1=bits-1, l2=0, fac=-1. The condition `i < l2` is `i < 0`, and starting at `i = bits-1` which is >= 0, so the loop iterates `bits-1, bits-2, ..., 0`. Actually no — `i < l2` with l2=0: the condition is `i < 0`. Starting at i=bits-1 which is > 0, so the loop body never executes! This is a bug in the inverted path.

Actually wait, let me re-read the condition more carefully:
```c
int l1 = 0, l2 = bits - rounds;
int fac = 1;
if (inverted) {
    l1 = bits - 1;
    l2 = 0;
    fac = -1;
}
for (int i = l1; i < l2; i += fac) {
```

When inverted: `for (int i = bits-1; i < 0; i -= 1)`. Since bits-1 >= 0, the condition `i < 0` is immediately false. **The loop never executes in inverted mode.** This means inverted CP_sequence is a no-op.

Check where inverted CP_sequence is called:
- QQ_mul block 1 (line 215): `CP_sequence(mul, &layer, rounds, 2*bits-1-bit, pow(2,bits)-1, false, bits)` — NOT inverted, OK
- all_rot (line 64): `CP_sequence(mul, layer, rounds, control, -multiplyer/2, inverted, bits)` — inverted parameter comes from all_rot's inverted parameter
- all_rot is called with `false` in QQ_mul block 2 (lines 226-227), and various true/false in cQQ_mul

Since inverted=false is the main path used, and the inverted path is a no-op (likely a bug but separate from what we're fixing), focus on the non-inverted path.

**Fix 3: QQ_mul block 2 (lines 220-237) — DIFFERENT pattern from QQ_add**

QQ_mul block 2 has a direct CP gate at line 235:
```c
cp(g, bits - i - 1, 3 * bits - bit_int2 - 1, value);
```

**IMPORTANT: This is NOT inside the QFT addition loop structure.** The target formula here is `bits - i - 1` (no `rounds` variable), which is structurally different from the `bits - i - 1 - rounds` pattern in QQ_add/CP_sequence. The fix is also different: target = `i`, NOT target = `rounds + i`.

**Why the derivation differs from QQ_add:**

In QQ_add (plan 29-09), each CP gate applies a single phase rotation from one control bit k to one target qubit j, with the inner loop stepping through target qubits for a fixed control bit. The target = `rounds + i` formula comes from the Draper control-target relationship (j = k + i).

QQ_mul block 2 is a **culmination optimization**. Instead of applying individual CP gates for each control-target pair (as QQ_add does), block 2 pre-accumulates the total phase that each target qubit j should receive from ALL control bits, then applies a SINGLE CP gate per target qubit. The `value` variable (lines 229-233) sums: `value += multiplier * pow(2, i - bit_int2)` across all control bit positions `bit_int2`. This `value` is the total accumulated rotation for target qubit position `i` in the QFT domain.

Since the target register is already in the QFT domain (QFT was applied earlier), and gate.c's QFT convention maps qubit index j directly to QFT position j (qubit 0 = first processed = highest frequency, as shown in gate.c lines 358-365), the culmination gate for QFT position `i` simply targets circuit qubit `i`.

There is no `rounds` variable because this is not iterating over control bits — the control bit iteration happens inside the `value` accumulation inner loop (lines 229-233), not in the CP gate application. The outer loop variable `i` directly indexes the target qubit in the QFT domain.

**The current `bits - i - 1` reverses this mapping** (i=0 -> qubit bits-1, i=bits-1 -> qubit 0), which is the same mirror-image error seen throughout the codebase. The correct target is simply `i`.

**Summary of the two different fix patterns:**
- QQ_add / CP_sequence: `bits - i - 1 - rounds` → `rounds + i` (Draper per-control-bit target)
- QQ_mul block 2: `bits - i - 1` → `i` (culmination gate, direct QFT qubit index)

**Change line 235:**
```c
cp(g, bits - i - 1, 3 * bits - bit_int2 - 1, value);
```
To:
```c
cp(g, i, 3 * bits - bit_int2 - 1, value);
```
