```bash

import quantum_language as ql
import qiskit.qasm3
from qiskit_aer import AerSimulator

def verify(op_name, builder_fn, expected, width=1):
    ql.circuit()
    builder_fn()
    qasm = ql.to_openqasm()
    circuit = qiskit.qasm3.loads(qasm)
    if not circuit.cregs:
        circuit.measure_all()
    sim = AerSimulator(method='statevector')
    job = sim.run(circuit, shots=1)
    counts = job.result().get_counts()
    bits = list(counts.keys())[0][:width]
    actual = int(bits, 2)
    status = 'PASS' if actual == expected else 'FAIL'
    print(f'{op_name}: expected={expected}, actual={actual} [{status}]')

# Test __lt__
verify('3 < 7', lambda: (ql.qint(3,width=4).__lt__(ql.qint(7,width=4))), 1)
verify('7 < 3', lambda: (ql.qint(7,width=4).__lt__(ql.qint(3,width=4))), 0)
verify('5 < 5', lambda: (ql.qint(5,width=4).__lt__(ql.qint(5,width=4))), 0)

verify('3 > 7', lambda: (ql.qint(3,width=4).__gt__(ql.qint(7,width=4))), 0)
```

Based on Task 1 diagnostics, apply the appropriate fix. There are three likely scenarios:

**Scenario A: __gt__ works correctly for qint-qint**
If `__gt__` produces correct results for all qint-qint cases (including 5>5=0, 7>3=1, 3>7=0), then the simplest fix is:

Replace the __le__ qint-qint block (lines ~1946-1967) with:
```python
# Handle qint operand
if type(other) == qint:
    return ~(self > other)
```

And replace the __le__ int block (lines ~1970-1994) with:
```python
# Handle int operand
if type(other) == int:
    # Classical overflow checks (keep existing)
    max_val = (1 << self.bits) - 1 if self.bits < 64 else (1 << 63) - 1
    if other < 0:
        return qbool(False)
    if other > max_val:
        return qbool(True)
    return ~(self > other)
```

IMPORTANT: The __gt__ int path currently calls `~(self <= other)` which creates a circular dependency. You MUST also fix __gt__ int path to avoid infinite recursion. Change __gt__ int path (line ~1897) from:
```python
return ~(self <= other)
```
to:
```python
# Create temp qint to use the qint-qint __gt__ path
temp = qint(other, width=self.bits)
return self > temp
```

This avoids the circular __gt__ -> __le__ -> __gt__ loop.

**Scenario B: __gt__ also fails for qint-qint**
If __gt__ also returns wrong values, the problem is in the subtraction + MSB extraction, not the OR logic. In this case, investigate:
1. Whether `other -= self` correctly computes the subtraction
2. Whether `other[64 - other.bits]` correctly extracts the MSB
3. Whether the qbool XOR (`result ^= msb`) correctly copies the bit

**Scenario C: __lt__ works but __gt__ fails (or vice versa)**
This indicates the subtraction direction matters. Compare the two implementations to find the difference.
