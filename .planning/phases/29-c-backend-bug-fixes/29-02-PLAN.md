**Diagnose BUG-04:** Read IntegerAddition.c focusing on the QFT-based addition functions.

Research identified two main functions:
- `CQ_add(bits)`: Classical value + quantum operand (uses classical `value` parameter)
- `QQ_add(bits)`: Quantum + quantum operand (both operands are qubits)

The bug manifests when **both operands are nonzero** (e.g., 3+5). Single-zero additions (0+N, N+0) may work correctly.

**Debug approach:**
1. Read IntegerAddition.c fully, focusing on:
   - CQ_add lines 49-56: rotation angle calculation `bin[bits - i - 1] * 2 * M_PI / (pow(2, j + 1))`
   - QQ_add lines ~132+: quantum-quantum addition implementation
   - Qubit layout comments (lines 23-30, lines 132-134)
2. Determine which function is actually used for qint + qint (check qint.pyx addition_inplace to see which C function gets called)
3. Test isolation: verify which cases fail:
   - 0 + 0 = 0 (both zero)
   - 0 + 1 = 1 (one zero)
   - 1 + 0 = 1 (other zero)
   - 1 + 1 = 2 (both nonzero, simplest case)
   - 3 + 5 = 8 (both nonzero, reported case)
4. Compare the QFT addition rotation formula against the standard textbook formula:
   - For QFT addition: after QFT on target, apply controlled phase rotations from control qubits
   - Phase angle for qubit j controlled by qubit k: 2*PI / 2^(j-k+1)
5. Check bin[] conversion (is the classical value correctly decomposed into bits?)
6. Check qubit offset/indexing (do gates target the right qubits?)

**Apply minimal fix** to IntegerAddition.c:
- Fix rotation angle formula if incorrect
- Fix qubit indexing if offset is wrong
- Fix bin[] decomposition if value-to-bits conversion is wrong
- Only change what's necessary -- QFT approach must remain (no fallback to ripple-carry per context)
