<objective>
Fix CQ_add's apparent asymmetry (0+X fails while X+0 works) and non-deterministic results caused by BUG-05 cache pollution.

Purpose: Verification shows CQ_add returns correct results for some inputs (1+0=1, 1+1=2) but wrong for others (0+1=2, 3+5=14). However, the mathematical analysis of the CQ_add algorithm shows the phase rotations ARE correct. The "asymmetry" is actually caused by BUG-05: the precompiled cache stores gate sequences from the FIRST call, then subsequent calls only update GateValue fields. If the first call creates the sequence structure with one set of layer/gate counts, those structure elements persist even when called with different values. This means: (1) The first call to CQ_add with value=V creates the structure, (2) A second call with value=W updates GateValue but the gate COUNT might be wrong if NonZeroCount differs.

Output: CQ_add with proper cache handling that prevents stale structural data from corrupting results.

**ROOT CAUSE ANALYSIS:**

The CQ_add function (lines 23-131) has a caching mechanism:
1. First call: allocates full sequence structure, computes QFT layers, adds phase gates, caches result
2. Subsequent calls: retrieves cached sequence, only updates GateValue fields (lines 65-73)

The cache update path (lines 68-70) does:
```c
add->seq[start_layer + i][add->gates_per_layer[start_layer + i] - 1].GateValue = rotations[i];
```

This updates the LAST gate in each layer. But the initial build path (lines 114-115) does:
```c
p(&add->seq[start_layer + i][add->gates_per_layer[start_layer + i]++], i, rotations[i]);
```

Note the `++` post-increment on `gates_per_layer`. After the initial build, `gates_per_layer[start_layer + i]` is 1 (was 0, incremented to 1). So the cache update path accesses index `1 - 1 = 0`, which is the same gate. This is correct.

**So the cache update itself is fine.** The issue must be elsewhere.

Let me re-examine the CQ_add phase computation for value=1, bits=4:
- bin = two_complement(1, 4) = [0, 0, 0, 1] (MSB-first)
- With the fix from plan 29-03: `bin[bits - 1 - bit_idx]`
  - bit_idx=0: bin[3] = 1 → contributes to rotations[0..3]
  - bit_idx=1: bin[2] = 0 → no contribution
  - bit_idx=2: bin[1] = 0 → no contribution
  - bit_idx=3: bin[0] = 0 → no contribution
- rotations[0] = 1 * 2*PI/2 = PI
- rotations[1] = 1 * 2*PI/4 = PI/2
- rotations[2] = 1 * 2*PI/8 = PI/4
- rotations[3] = 1 * 2*PI/16 = PI/8
- Phase gates: P(qubit=0, PI), P(qubit=1, PI/2), P(qubit=2, PI/4), P(qubit=3, PI/8)

For adding +1 to a 4-bit number in QFT domain (with qubit 0 = first QFT qubit):
- qubit 0 gets exp(2PI*i * 1/2) → phase = PI ✓
- qubit 1 gets exp(2PI*i * 1/4) → phase = PI/2 ✓
- qubit 2 gets exp(2PI*i * 1/8) → phase = PI/4 ✓
- qubit 3 gets exp(2PI*i * 1/16) → phase = PI/8 ✓

**The algorithm is MATHEMATICALLY CORRECT.** So why does 0+1 return 2?

The answer is likely BUG-05 (cache pollution between tests) or a qubit readout ordering issue. Since 0+1=2 means the result is binary 0010 instead of 0001 (off by one bit position), this strongly suggests the qubit layout convention between CQ_add and the readout is mismatched by one bit position.

**ALTERNATIVE DIAGNOSIS: Check if the issue is in how the test initializes qint(0).**

When `a = ql.qint(0, width=4)` is created and then `a += 1` is called:
- qint(0) initializes all qubits to |0>
- a += 1 calls CQ_add(bits=4, value=1)
- CQ_add applies QFT, then phase rotations, then inverse QFT
- The result should be the state |0001> (value 1)

If the result is 2 (|0010>), the QFT domain mapping is off by one position. Check if the start_layer calculation is correct. Currently `start_layer = bits` (line 62). But QFT uses `2 * num_qubits - 1` layers (line 367 of gate.c: `qft->used_layer += 2 * num_qubits - 1`). So after QFT, `add->used_layer` should be `2*bits - 1`.

**CRITICAL BUG FOUND:** After QFT, used_layer = 2*bits - 1. But `start_layer = bits` (line 62). This means the phase gates start at layer `bits`, but QFT ends at layer `2*bits - 2` (0-indexed). The phase gates at layers bits through 2*bits-1 would OVERLAP with the QFT layers!

Wait, let me re-read. QFT sets `qft->used_layer += 2 * num_qubits - 1`. The initial used_layer is 0. So after QFT, used_layer = 2*bits-1. The QFT occupies layers 0 through 2*bits-2.

The phase rotation start_layer = bits. For bits=4: start_layer=4, QFT uses layers 0..6. So the phase gates at layers 4,5,6,7 would overlap with QFT layers 4,5,6 and extend to layer 7!

**But wait** — the sequence was allocated with `num_layer = 4 * bits - 1` total layers. And the phase gates need bits layers after QFT. QFT uses 2*bits-1 layers, then phase rotations need bits layers, then inverse QFT needs 2*bits-1 more. Total = 2*(2*bits-1) + bits = 5*bits - 2. But the allocation is only 4*bits - 1 layers!

Hmm, but QQ_add allocates `5 * bits - 2` layers (line 156). CQ_add allocates `4 * bits - 1` layers (line 83). This is because CQ_add's phase rotations are just single-qubit P gates (1 gate per target qubit, all on different qubits, so they can share layers with QFT).

Actually, re-examining: the gates are being placed at specific layer indices using `start_layer + i`. The QFT and inverse QFT also place gates at specific indices. The point is that `start_layer = bits` means the phase gates go into layers `bits, bits+1, ..., 2*bits-1`. These are the SAME layers as the latter half of the QFT. But those layers may already have QFT gates on different qubits — and since each layer can hold multiple gates, this is intentional (gates on different qubits in the same layer are parallel).

Let me look more carefully at what qubit each QFT gate targets vs what qubit the phase gate targets. QFT layer `2*j + k + 1` has a CP on qubit `j` controlled by qubit `j+k+1`. For j=0, layers 1,2,3 have CPs on qubit 0. For j=1, layer 2 has H on qubit 1, layers 3,4 have CPs on qubit 1. For j=2, layer 4 has H on qubit 2, layer 5 has CP on qubit 2. For j=3, layer 6 has H on qubit 3.

Phase gate at start_layer + i = bits + i targets qubit i. For bits=4: layer 4 targets qubit 0, layer 5 targets qubit 1, layer 6 targets qubit 2, layer 7 targets qubit 3.

QFT layer 4: H on qubit 2 + CP(qubit 1, control qubit 3). Phase gate at layer 4: P(qubit 0). These are on different qubits, so they CAN coexist in the same layer.

Layer 5: QFT has CP(qubit 2, control qubit 3). Phase gate: P(qubit 1). Different qubits, OK.

Layer 6: QFT has H on qubit 3. Phase gate: P(qubit 2). Different qubits, OK.

Layer 7: No QFT gate. Phase gate: P(qubit 3). OK.

So the overlapping is intentional and correct — the gates act on different qubits.

**Then after the phase rotations, `add->used_layer++` (line 118). This sets used_layer to 2*bits-1 + 1 = 2*bits. Then QFT_inverse starts appending from there.**

Actually no — let me re-read line 118: `add->used_layer++;`. But what was used_layer before this? After QFT (line 111), used_layer = 2*bits-1. The phase rotation loop doesn't change used_layer (it just writes into existing layers). Then `add->used_layer++` sets it to 2*bits. Then QFT_inverse (line 120) will start from layer 2*bits.

But the phase gates were placed at layers bits to 2*bits-1. That's within the QFT layers (0 to 2*bits-2). Then used_layer is set to 2*bits. QFT_inverse starts at 2*bits.

Wait, I think the `add->used_layer++` is wrong. After QFT, used_layer = 2*bits-1. The code writes phase gates into layers bits..2*bits-1 (these overlap with QFT layers, which is fine for parallelism). But then the code increments used_layer from 2*bits-1 to 2*bits. The inverse QFT will start at layer 2*bits. Total layers used = 2*bits + (2*bits-1) = 4*bits-1. This matches the allocation `4*bits - 1`.

So structurally this seems correct. The phase gates share layers with the end of the QFT, and everything fits.
