<objective>
Fix CQ_add's apparent asymmetry (0+X fails while X+0 works) and non-deterministic results caused by BUG-05 cache pollution.

Purpose: Verification shows CQ_add returns correct results for some inputs (1+0=1, 1+1=2) but wrong for others (0+1=2, 3+5=14). Mathematical analysis confirms the CQ_add phase rotation algorithm is correct. The root cause is BUG-05: the precompiled cache stores gate sequences from the FIRST call, then subsequent calls only update GateValue fields. If the first call creates the structure with one NonZeroCount, that structure persists even when called with different values where NonZeroCount differs.

Output: CQ_add with NonZeroCount-aware cache invalidation that prevents stale structural data from corrupting results.
</objective>

**Algorithm correctness analysis (for reference -- do NOT re-derive, this is settled):**

The CQ_add algorithm is mathematically correct:
- CQ_add phase computation for value=1, bits=4: rotations = [PI, PI/2, PI/4, PI/8]
- Phase gates: P(qubit=0, PI), P(qubit=1, PI/2), P(qubit=2, PI/4), P(qubit=3, PI/8)
- These match the Draper QFT adder requirements exactly
- The layer structure (start_layer=bits, overlapping with QFT tail) is correct because phase gates and QFT gates target different qubits in shared layers
- Total layers = 4*bits-1 matches the allocation

The 0+1=2 failure is caused by cache pollution (BUG-05), not an algorithm bug. The cache update path (lines 68-70) correctly updates GateValue at index 0 for each layer. However, when NonZeroCount changes between calls, the cached structure has wrong gate counts.
</context>
