# Phase 113: Diffusion Redesign & Move Enumeration - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the walk diffusion operator to handle large branching factors (24+ children) without combinatorial explosion, and create a fixed move enumeration table for quantum evaluation. The old itertools.combinations approach in walk.py and chess_walk.py is replaced entirely with an arithmetic counting circuit. All-moves enumeration provides the fixed branch indices that the quantum legality predicate (Phase 114-115) will evaluate.

</domain>

<decisions>
## Implementation Decisions

### Counting circuit design
- Use qarray sum reduction (`sum(validity_arr)`) to sum validity bits into a count register
- Count register width computed dynamically: `ceil(log2(d_max + 1))` qubits — minimal allocation, adapts to any branching factor
- Wrap counting logic in `@ql.compile(inverse=True)` for clean forward/inverse lifecycle: compute count, do diffusion rotations, call `.inverse()` to uncompute

### Diffusion angle dispatch
- Compare-and-rotate loop: iterate d from 1 to d_max, compare `count == d`, then controlled-Ry(phi(d)) — O(d_max) iterations, each O(w) gates
- S_0 reflection stays unchanged: `ql.diffusion(h_child, branch_reg)` controlled on h[depth] — independent of d(x)
- Cascade rotations still operate on branch register qubits, controlled by count comparison flag
- Validity-gated cascade: only rotate into branch states for valid children, using `with validity[i]:` as control on each cascade Ry

### Move enumeration table
- Fixed move indices based on (piece_instance, geometric_offset) — NOT a flat list of all (src, dst) pairs
- d_max = 8 per piece x number of pieces: white KNK = 24 (K:8 + N1:8 + N2:8), black K = 8
- Python precomputation function: takes classically known piece configuration (which piece types remain per side), returns move table mapping branch index to (piece_id, dx, dy offset)
- Piece positions are in superposition; piece configuration (which types exist) is classically known
- Table is a Python-level data structure (not quantum-encoded) — oracle reads it at circuit construction time
- Separate tables per side: white and black have independent move tables with different sizes and branch register widths

### Backward compatibility
- Replace old combinatorial diffusion entirely in both walk.py and chess_walk.py — no coexistence
- chess_walk.py shares the new diffusion from walk.py (single implementation, no duplication)
- Verify equivalence via statevector comparison on small cases (d_max=2, 3, 4) before removing old code
- Existing walk tests (SAT demo, reflection property tests) must pass with the new counting-based diffusion

### Claude's Discretion
- Exact qarray sum reduction implementation details (handling of 1-bit width elements)
- Cascade angle computation internals (reuse existing `_plan_cascade_ops` or redesign)
- Module organization for the shared diffusion function (method on QWalkTree vs standalone)
- How the move table builder handles edge-of-board filtering for geometric offsets

</decisions>

<specifics>
## Specific Ideas

- User initially considered d_max=218 (max legal chess moves) but realized classical move indexing doesn't translate to quantum superposition — led to the piece+offset decomposition
- Move table should be generated by a Python function that takes remaining pieces per color as input — this is a precomputation step before circuit construction
- The framework's own constructs (qarray sum, @ql.compile inverse, with qbool, qint ==) should be used throughout — no raw gate emission for the counting or dispatch logic

</specifics>

- `QWalkTree._variable_diffusion()` (walk.py:782): Current combinatorial diffusion — structure to be replaced but angle computation and S_0 logic reusable
- `precompute_diffusion_angles()` (chess_walk.py:214): Montanaro angle computation — reusable as-is

- `@ql.compile(inverse=True)`: Existing compilation infrastructure for forward/inverse lifecycle

- `knight_attacks()` / `king_attacks()` (chess_encoding.py): Geometric offset tables — basis for move table builder

### Established Patterns
- V-gate CCRy decomposition for multi-controlled Ry (avoids nested `with qbool:` limitation)
- `_emit_multi_controlled_ry()` and `_emit_cascade_multi_controlled()` for controlled rotations
- Factory pattern for compiled oracles (chess_encoding.py)

### Integration Points
- walk.py `_variable_diffusion()`: Primary replacement target
- chess_walk.py `apply_diffusion()`: Secondary replacement target — will call shared function from walk.py
- chess_walk.py `evaluate_children()` / `uncompute_children()`: Unchanged — still allocate validity qbools used by counting circuit
- Phase 114-115 predicates: Will use the move table indices to know which piece and offset to evaluate
