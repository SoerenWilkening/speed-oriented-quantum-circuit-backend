<objective>
Fix two independent bugs that prevent indexed branch and controlled branch tests from passing:

1. **__getitem__ wrong offset** (src/quantum_language/qint_bitwise.pxi:726): `self.qubits[item]` uses the raw logical index into the 64-element qubit array, but qubits are stored RIGHT-ALIGNED at indices `[64-width]` through `[63]`. For a 3-qubit qint, `x[0]` reads `self.qubits[0]` which is uninitialized memory — the LSB is at `self.qubits[61]`. The Ry gate targets a garbage qubit index, leaving the actual qubits untouched.

2. **Wrong Qiskit bitstring convention in test** (tests/python/test_branch_superposition.py:221-226): Qiskit uses little-endian convention (q[0] is rightmost bit). `ctrl` is allocated as q[0] (LSB) and `target` as q[1] (MSB). The test incorrectly assumes ctrl is the MSB and checks bitstrings "10"/"11". The correct bitstrings for ctrl=|1> states are "01" (ctrl=1, target=0) and "11" (ctrl=1, target=1).

  <files>src/quantum_language/qint_bitwise.pxi</files>
  <action>
    In `qint.__getitem__()` (line 726), the current line is:

    ```python
    bit_list[-1] = self.qubits[item]
    ```

    This is wrong. `item` is the logical index (0 = LSB), but qubits are stored right-aligned in a 64-element array. For a qint with `self.bits` width, the LSB (index 0) lives at `self.qubits[64 - self.bits + 0]`, and the MSB (index `bits-1`) lives at `self.qubits[63]`.

    Reading `self.qubits[0]` (or any `self.qubits[item]` where item < 64-self.bits) reads uninitialized memory. The gate emitted by branch() then targets a garbage qubit index, having no effect on the actual allocated qubits.

    Change line 726 to:

    ```python
    bit_list[-1] = self.qubits[64 - self.bits + item]
    ```

    Also add a bounds check immediately before the array access (after the `from quantum_language.qbool import qbool` line, before `bit_list = np.zeros...`):

    ```python
    if item < 0 or item >= self.bits:
    ```

  <name>Task 2: Fix Qiskit bitstring convention in TestBranchControlled</name>
  <files>tests/python/test_branch_superposition.py</files>
  <action>
    In `TestBranchControlled.test_controlled_branch_cry()` (lines 219-226), the test currently checks:

    ```python
    # Expected: |11> and |10> with equal probability (target in superposition)
    # ctrl is always 1 (MSB), target can be 0 or 1
    p_10 = counts.get("10", 0) / total
    p_11 = counts.get("11", 0) / total

    assert p_10 + p_11 > 0.95, f"Control=|1> so target should be active"
    assert abs(p_10 - 0.5) < 0.05
    assert abs(p_11 - 0.5) < 0.05
    ```

    This is wrong. Qiskit uses little-endian bitstring ordering: for a 2-qubit circuit, bitstring "ab" means q[1]=a (leftmost), q[0]=b (rightmost). Since `ctrl = ql.qbool(True)` is allocated first (q[0] = LSB) and `target = ql.qbool(False)` second (q[1] = MSB):

    - "10" means q[1]=1, q[0]=0 → target=1, ctrl=0 (ctrl NOT set — wrong states to check)
    - "01" means q[1]=0, q[0]=1 → target=0, ctrl=1 (ctrl IS set, target=0)
    - "11" means q[1]=1, q[0]=1 → target=1, ctrl=1 (ctrl IS set, target=1)

    The correct expected states when ctrl=|1> are "01" and "11". Replace the comment and assertions:

    ```python

    # ctrl is q[0] (LSB, rightmost), target is q[1] (MSB, leftmost)
    # ctrl=|1> states: "01" (target=0, ctrl=1) and "11" (target=1, ctrl=1)
    p_01 = counts.get("01", 0) / total
    p_11 = counts.get("11", 0) / total

    assert p_01 + p_11 > 0.95, f"Control=|1> so target should be active. Got: {counts}"
    assert abs(p_01 - 0.5) < 0.05, f"Expected p_01~0.5, got {p_01:.4f}"
    assert abs(p_11 - 0.5) < 0.05, f"Expected p_11~0.5, got {p_11:.4f}"
    ```

    Read test_branch_superposition.py lines 195-227 before editing to confirm current code.
