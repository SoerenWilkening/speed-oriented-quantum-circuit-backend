# Architecture Research: Function Compilation (@ql.compile Decorator)

**Domain:** Quantum circuit compilation/caching for AriaQuanta
**Researched:** 2026-02-04
**Confidence:** HIGH (based on thorough codebase analysis)

## Executive Summary

The `@ql.compile` decorator must intercept the gate-generation pipeline at the Python/Cython boundary, capture a gate sequence during the first invocation, optimize it, and replay it (with qubit remapping) on subsequent calls. The existing architecture provides strong primitives for this: `run_instruction()` already performs qubit remapping from template sequences, `circuit_optimize()` copies circuits through `add_gate()` for optimization, and the `layer_floor` mechanism controls gate placement boundaries. The main architectural challenge is managing the gap between Python-level objects (qint with qubit references) and the C-level gate stream, especially for parametric classical values and controlled (`with` block) contexts.

## Current Architecture (Relevant Parts)

### Gate Flow: Python Operation to Circuit

The complete data flow for a quantum operation like `a + b`:

```
Python: a + b
  -> qint.__add__(self, other)
    -> allocates result qint (calls allocator_alloc for qubits)
    -> copies self into result via XOR
    -> calls addition_inplace(other)
      -> checks _controlled flag (for `with` block context)
      -> packs qubit indices into qubit_array[] (numpy uint32 array)
      -> selects sequence function: QQ_add(bits) or cQQ_add(bits) if controlled
      -> calls run_instruction(seq, qubit_array, invert, _circuit)

C: run_instruction(sequence_t *seq, qubit_array[], invert, circuit_t *circ)
  -> iterates layers in seq (forward or reverse if inverted)
    -> for each gate in layer:
      -> copies gate_t, remaps Target and Control[] via qubit_array[]
      -> calls add_gate(circ, g)

C: add_gate(circuit_t *circ, gate_t *g)
  -> allocates qubit/layer storage as needed
  -> finds minimum_layer() respecting circ->layer_floor
  -> checks colliding_gates() for inverse cancellation (merge_gates)
  -> if not cancelled: append_gate() + apply_layer()
```

**Key insight:** Sequences (`sequence_t`) are pre-built gate templates with abstract qubit indices (0, 1, 2...) that `run_instruction()` maps to actual qubit indices. This is exactly the pattern needed for compiled function replay.

### Circuit Structure

```c
circuit_t {
    gate_t **sequence;           // [layer][gate_index] - the actual gate storage
    num_t used_layer;            // current depth
    num_t layer_floor;           // minimum layer for new gates (set by Python)
    qubit_allocator_t *allocator; // qubit lifecycle management
}

gate_t {
    qubit_t Target;
    qubit_t Control[MAXCONTROLS]; // up to 2 inline
    qubit_t *large_control;       // >2 controls heap-allocated
    num_t NumControls;
    Standardgate_t Gate;          // X, Y, Z, H, P, etc.
    double GateValue;             // rotation angle for P/R gates
}
```

### How `with` Blocks Work (Controlled Context)

The `with` statement on a qbool sets global flags in `_core.pyx`:

```python
# qint.__enter__():
_set_controlled(True)
_set_control_bool(self)  # the qbool acting as control

# Each operation checks:
if _controlled:
    seq = cCQ_add(bits, value)  # controlled variant
    qubit_array[start] = control_bool.qubits[63]  # pack control qubit
else:
    seq = CQ_add(bits, value)   # uncontrolled variant
```

The controlled variants are entirely different `sequence_t` objects generated by the C backend (e.g., `cCQ_add` vs `CQ_add`). The control qubit is packed into the qubit_array and mapped by `run_instruction()`.

### How Optimization Works

Two optimization stages exist:

1. **During construction** (`add_gate()` in optimizer.c): Checks if new gate is inverse of the gate at the previous layer on the same qubit. If so, merges (cancels) them. This happens automatically for every gate.

2. **Post-construction** (`circuit_optimize()` in circuit_optimizer.c): Creates a new circuit by replaying all gates through `add_gate()`, which re-applies the inverse cancellation. This is the "copy-circuit" pattern.

### Qubit Allocation

Qubits are allocated sequentially from a `qubit_allocator_t` attached to the circuit. Each `qint` tracks:
- `allocated_start`: starting qubit index
- `bits`: width
- `qubits[64]`: right-aligned array of actual qubit indices

Deallocation (via uncomputation) returns qubits to the allocator for reuse.

### Layer Floor Mechanism

The `layer_floor` field in `circuit_t` (set by Python before calling `run_instruction()`) tells `add_gate()` the minimum layer at which new gates can be placed. This is used by operations like `__add__` to prevent the optimizer from scheduling gates before the operation's "start" point. The `_start_layer`/`_end_layer` fields on qint track the layer range of an operation for uncomputation.

## Proposed Integration

### Design: "Capture-Optimize-Replay" at the Circuit Level

The decorator should work at the **circuit gate level** rather than the Python object level. This avoids needing to understand Python semantics -- it simply records what gates were added.

#### Phase 1: Capture

On first call of a `@ql.compile`-decorated function:
1. Record the circuit's `used_layer` before the function runs (start_layer)
2. Execute the function normally (all gates flow through `add_gate()` as usual)
3. Record `used_layer` after execution (end_layer)
4. Extract the gate range `[start_layer, end_layer)` as a `compiled_block_t`
5. Record the qubit mapping: which abstract parameter positions map to which actual qubits

#### Phase 2: Optimize

Optimize the captured block:
- Create a temporary `circuit_t` containing only the captured gates
- Run `circuit_optimize()` on it (or apply more aggressive passes)
- Store the optimized gate list as the compiled template

#### Phase 3: Replay

On subsequent calls:
1. Map the caller's qint qubit indices to the template's abstract qubit positions
2. Inject optimized gates into the current circuit using `run_instruction()`-style qubit remapping
3. Respect the current `layer_floor` and controlled context

### Core Data Structures

```python
# Python level (new):

class CompiledBlock:
    """Captured and optimized gate sequence from a @ql.compile function."""
    gates: list[dict]          # list of {type, target_idx, controls[], angle}
    abstract_qubit_count: int  # total abstract qubits used
    param_qubit_ranges: list[tuple[int, int]]  # (start, width) per parameter
    internal_qubit_count: int  # qubits allocated internally (temporaries)
    return_qubit_range: tuple[int, int] | None  # abstract qubits for return value

class CompiledFunc:
    """Wrapper around a @ql.compile decorated function."""
    func: callable             # original Python function
    cache: dict                # (classical_args, widths, controlled) -> CompiledBlock
```

At the C level, no new structures are needed for Phase 1. An optional `compiled_sequence_t` compatible with `run_instruction()` could be added in a performance-focused Phase 2.

### Handling the Six Key Questions

#### 1. Gate Capture: Layer Range Extraction

Use the same pattern already used by `_start_layer`/`_end_layer` tracking in qint operations:

```python
start_layer = get_current_layer()  # already exposed in _core.pyx
result = func(*args)  # normal execution - gates flow to global circuit
end_layer = get_current_layer()
captured = extract_gate_range(circuit, start_layer, end_layer)
```

This is the simplest approach and requires no changes to `add_gate()`. The existing `layer_floor` mechanism (already used by `__add__`, `__mul__`, etc.) ensures captured gates are placed in a predictable range.

**Alternative considered and rejected:** Redirecting to a separate circuit buffer would require changing the global `_circuit` pointer, which would break qubit allocation (the allocator is tied to the circuit). The layer-range approach avoids this entirely.

**New Cython function needed:** `extract_gate_range(start, end)` that reads `circuit_t.sequence[start:end]` and returns a Python list of gate dicts. This mirrors the existing `draw_data()` method which already iterates all gates in the circuit and converts them to Python dicts.

#### 2. Parametric Classical Values

Classical parameters (e.g., `value` in `CQ_add(bits, value)`) determine which `sequence_t` is generated by the C backend. Different classical values produce different gate sequences.

**Recommended approach:** Cache by classical parameter values and widths.

```python
cache_key = (classical_arg_tuple, width_tuple, is_controlled)
```

Quantum parameters (qint/qbool) only affect qubit remapping, not the gate structure, so they do not need to be part of the cache key. Width must be part of the key since different widths produce different sequences (e.g., `CQ_add(8, 5)` vs `CQ_add(16, 5)` are structurally different).

#### 3. Replay: Injecting Pre-Built Gates

The replay mechanism mirrors `run_instruction()`:

```python
def replay(compiled_block, qubit_args, circuit_ptr):
    # Build qubit mapping: abstract_idx -> actual_qubit
    mapping = {}
    for param_idx, qint_arg in enumerate(qubit_args):
        start, width = compiled_block.param_qubit_ranges[param_idx]
        for bit_idx in range(width):
            mapping[start + bit_idx] = qint_arg.qubits[64 - width + bit_idx]

    # Allocate internal temporaries (fresh qubits for this call)
    for i in range(compiled_block.internal_qubit_count):
        abstract_idx = compiled_block.internal_start + i
        mapping[abstract_idx] = allocator_alloc(allocator, 1, True)

    # Inject gates with remapping (Cython function calling C add_gate)
    inject_remapped_gates(compiled_block.gates, mapping, circuit_ptr)

    # Build return qint referencing the appropriate mapped qubits
    if compiled_block.return_qubit_range is not None:
        return build_return_qint(compiled_block.return_qubit_range, mapping)
```

This can be implemented at the Python/Cython level initially, calling the existing C `add_gate()` for each remapped gate. A future C-level `inject_remapped_gates()` would be faster but is not required for correctness.

#### 4. Working Inside `with` Blocks

When a compiled function is called inside a `with` block, every gate needs the control qubit added. Two approaches:

**Approach A (Recommended for Phase 1): Re-capture with control.** On first call inside a `with` block, the `_controlled` flag is already set globally. The Python-level operations automatically select controlled sequence variants (`cCQ_add` etc.). So the capture produces a controlled gate sequence naturally. Cache this separately:

```python
cache_key = (classical_args, widths, is_controlled, num_controls)
```

**Approach B (Future optimization): Post-hoc control addition.** Add control qubits to every gate in the compiled block during replay. This would require modifying each gate's `Control[]` and `NumControls` during injection. This is more complex but avoids full re-execution for the controlled case.

Approach A is recommended for Phase 1 because it requires no new C code and leverages the existing controlled-variant infrastructure. The overhead of re-capture is acceptable since it happens only once per (args, controlled) combination.

#### 5. Return Values: Qubit Ownership Transfer

When a compiled function returns a qint, the caller needs a Python qint object whose `.qubits` array points to the correct physical qubits allocated during replay.

During replay:
1. Internal qubits are allocated fresh from the allocator
2. A new `qint(create_new=False, bit_list=mapped_return_qubits)` is created
3. Ownership metadata (`allocated_start`, `allocated_qubits`, `_start_layer`, `_end_layer`) is set on the returned qint

This matches the existing pattern used by `qbool.copy()` (in `qbool.pyx` lines 76-88) which transfers qubit ownership between qint objects by setting `allocated_start`, `allocated_qubits=True` on the destination and `allocated_qubits=False` on the source.

**Important detail:** The returned qint must have correct `_start_layer` and `_end_layer` for uncomputation to work. During replay, these should span the injected gate range.

#### 6. Compilation Cache Location

**Python-level dict** is the right choice:

```python
# Per-decorated-function cache, stored as attribute on the wrapper
_cache: dict[tuple, CompiledBlock] = {}

# Key: (func_id, classical_args_tuple, param_widths_tuple, is_controlled)
# Value: CompiledBlock (Python object holding gate list + metadata)
```

Reasons:
- Cache invalidation tied to circuit resets (`circuit()` constructor resets everything)
- Classical args are Python objects (natural dict keys)
- No C memory management complexity
- The hot path (gate replay via `add_gate()`) is already in C

The cache must be cleared when `circuit()` is called (new circuit = new compilation context). This can be hooked into the existing `circuit.__init__()` which already resets all global state.

  y = ql.qint(5, width=8)

  r1 = my_op(x, y)   # FIRST CALL:
                      #   1. Record start_layer = circuit.used_layer

## Key Architectural Decisions

### Decision 1: Layer-range extraction vs. separate circuit buffer

**Chosen:** Layer-range extraction.

**Rationale:** Using a separate circuit buffer would require:
- Swapping the global `_circuit` pointer (breaks qubit allocator continuity)
- Managing two allocators simultaneously
- Ensuring circuit state consistency during capture

Layer-range extraction uses the existing single circuit, reads back the gates after execution, and requires zero changes to the C backend. The precedent exists: `_start_layer`/`_end_layer` tracking on every qint operation already records layer ranges for uncomputation.

### Decision 2: Python-level cache vs. C-level cache

**Chosen:** Python-level dict.

**Rationale:** The cache key includes Python-level information (function identity, classical arg values, width tuples). The hot path is gate injection (C-level `add_gate()`), which is already fast. Python dict lookup overhead is negligible compared to gate generation. Moving the cache to C would add complexity without meaningful performance gain.

### Decision 3: Re-capture for controlled variants vs. post-hoc control addition

**Chosen:** Re-capture for controlled variants (Phase 1).

**Rationale:** The existing architecture uses entirely different `sequence_t` generators for controlled vs uncontrolled operations (e.g., `CQ_add` vs `cCQ_add`). These produce structurally different gate decompositions, not simply "the same gates plus a control." Re-capture naturally handles this complexity. Post-hoc control addition would only work for simple single-qubit gates, not for the complex multi-gate sequences used in arithmetic.

### Decision 4: How to handle qubit allocation during replay

**Chosen:** Allocate fresh internal qubits, remap parameter qubits from caller.

**Rationale:** During replay, the function's internal temporaries need fresh qubits (different addresses than the original capture). Parameter qubits come from the caller's qint objects and are remapped. This mirrors exactly what `run_instruction()` does with `qubit_array[]` -- abstract template indices mapped to actual circuit qubits.

### Decision 5: Where in the three-layer architecture to implement

**Chosen:** Python decorator + Cython extraction/injection. No C changes.

**Rationale:** Compilation is orchestration logic (when to capture, when to replay, cache management). This is best expressed in Python. The performance-critical parts (reading/writing gates) are thin Cython functions calling existing C code. No new C data structures or algorithms are needed.

## Risk Analysis

| Risk | Severity | Mitigation |
|------|----------|------------|
| Gate extraction changes circuit state | HIGH | Extract is read-only (iterate `sequence[][]`, copy gate data) |
| Qubit allocator divergence between capture and replay | MEDIUM | Replay allocates fresh qubits via existing allocator, does not reuse captured addresses |
| `with` block state not properly captured | MEDIUM | Phase D: re-capture under controlled context; cache key includes controlled flag |
| Cache grows unbounded | LOW | Clear on `circuit()` reset (already resets all state); add LRU if needed |
| Nested `@ql.compile` calls | MEDIUM | Capture naturally handles nesting: inner compiled function gates are part of outer capture range |
| Functions with Python-level side effects | LOW | Document: `@ql.compile` functions should be pure quantum operations. Side effects only execute on first call. |
| Optimizer reorders gates differently on replay | LOW | Gates injected via `add_gate()` go through same optimizer; layer_floor prevents unwanted reordering |
| Internal temporary qubit count varies between calls | LOW | Count is fixed for given (widths, classical_args) -- same function, same structure |

## Existing Patterns to Leverage

### Pattern: `draw_data()` -- Reading Circuit Gates into Python

The existing `draw_data()` method in `_core.pyx` (lines 413-461) already iterates all gates in the circuit and converts them to Python dicts. The `extract_gate_range()` function follows the same pattern but for a layer subset.

### Pattern: `run_instruction()` -- Qubit Remapping

The C function `run_instruction()` in `execution.c` (lines 14-53) demonstrates the exact remapping pattern needed: iterate template gates, remap `Target` and `Control[]` via a mapping array, call `add_gate()`.

### Pattern: `reverse_circuit_range()` -- Layer Range Iteration

The C function `reverse_circuit_range()` in `execution.c` (lines 57-104) iterates a specific layer range of the circuit in reverse order, creating gate copies. This demonstrates how to iterate a subset of the circuit safely.

### Pattern: `qbool.copy()` -- Qubit Ownership Transfer

The `qbool.copy()` method (qbool.pyx lines 76-88) shows how to create a new qint that references existing qubits, transferring ownership metadata (`allocated_start`, `allocated_qubits`).

### Pattern: `circuit_optimize()` -- Copy-Through-AddGate

The `copy_circuit()` helper in `circuit_optimizer.c` (lines 17-34) creates a new circuit by replaying gates through `add_gate()`. This is conceptually the same as what compiled block optimization will do.
